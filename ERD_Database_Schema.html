2. List of Entities & Attributes

Entity 1: USERS

Represents all system users regardless of role. A single unified table stores seekers, employers, and admins, distinguished by the role field. Fields like company_name are used only by employers; nid_number / nid_image are used by seekers and employers during registration.

Represents job postings created by employers. Each job is linked to exactly one employer (via employer_id) and can receive many applications. Jobs have a lifecycle status: draft → active → closed.

Entity 3: APPLICATIONS

Acts as a junction/associative entity resolving the many-to-many relationship between USERS (seekers) and JOBS. Each row represents one seeker's application to one job. A unique constraint on (job_id, seeker_id) prevents duplicate applications.

3. Relationships & Cardinality

Relationship 1: USERS — POSTS — JOBS

ONE-TO-MANY (1:N) One employer (User) can post many Jobs, but each Job belongs to exactly one employer.

Participation: Total participation on the Jobs side (a job must have an employer); partial participation on the Users side (not all users post jobs — only employers do).

Implementation: jobs.employer_id → users.id with ON DELETE CASCADE (deleting an employer removes all their job postings).

Relationship 2: USERS — APPLIES TO — APPLICATIONS

ONE-TO-MANY (1:N) One seeker (User) can submit many Applications, but each Application belongs to exactly one seeker.

Participation: Total participation on the Applications side (an application must have a seeker); partial on the Users side (not all users apply — only seekers do).

Implementation: applications.seeker_id → users.id with ON DELETE CASCADE.

Relationship 3: JOBS — CONTAINS — APPLICATIONS

ONE-TO-MANY (1:N) One Job can receive many Applications, but each Application is for exactly one Job.

Participation: Total on the Applications side (every application must reference a job); partial on the Jobs side (a job may have zero applications).

Implementation: applications.job_id → jobs.id with ON DELETE CASCADE.

Note on Many-to-Many: The implicit many-to-many relationship between Users (seekers) and Jobs — "a seeker can apply to many jobs, and a job can receive applications from many seekers" — is resolved through the APPLICATIONS associative/junction entity, decomposing it into two 1:N relationships. The composite UNIQUE constraint on (job_id, seeker_id) enforces that each seeker can apply to a given job only once.
